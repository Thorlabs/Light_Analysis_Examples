This is the description of the Thorlabs PM103E remote interface. This single channel Power/Energy meter can be remotely controlled using the SCPI (Standard Commands for Programmable Instruments) interface.

In 1975, the IEEE standardized a bus developed by Hewlett-Packard originally called HPIB (Hewlett-Packard Interface Bus), later changed to GPIB (General Purpose Interface Bus). The standard was called IEEE 488 (IEEE 488.1) and it defined mechanical aspects of the bus. The later IEEE 488.2 defined its protocol properties. What was missing were set of rules between the manufacturers on commands to control the instruments. Sometimes these varied even between different models from the same manufacturer.

In 1990 SCPI Consortium released the first SCPI standard as an additional layer for the IEEE-488.2 standard.

SCPI commands are ASCII strings, which are sent to instrument over the physical communication layer. They can perform:

  • Set operations, for example the *RST command (resetting the instrument).
  • Query operations, for example the *IDN? query (querying the instrument’s identification string).

Some SCPI commands exist as both set commands and query commands. An example is an Powermeter's averaging command SENS:AVER. You can set it with the SCPI command SENS:AVER 2, and also query its current value with the SENS:AVER?.

The format mentioned in this manual e.g.: SENSe#:AVERage[:COUNt] <pres> is called canonical form. Here are the most important rules to remember:

  • The parts within square brackets are not mandatory and can be left out.
  • All # within the command represent a suffix positive integer number starting at 1. If there is only one suffix you may skip it because default is applied internally. If there are two or more suffix it is mandatory to specify all.
  • The capital letter parts are mandatory; the small letters can be omitted. This is called short form. An example of the above command in the short form is SENS1:AVER 3. You can use either the short form, or the long form SENSE1:AVERAGE 3, but nothing in between, e.g. SENS1:AVERA 3.
  • The SCPI commands are case-insensitive. You can also use the short form sens1:aver 3 or or the long form sense1:average 3
  • Combine multiple commands into one string using selmicolon ';'. For example, a combined string of SENS1:AVER 3 and SENS1:CORR:WAV 850 is SENS1:AVER 3;CORR:WAV 850. Notice that the second command does not have the SENS1: part. The reason is, that the command tree path does not change within one string. If you want to reset the command tree path to the root, use the colon character at the beginning of the second command: SENS1:AVER 3;:SENS2:CORR:WAV 850.
  • Create query forms by adding a question mark, mostly to the end: SENS1:AVER? Sometimes there is an additional parameter placed after the question mark. There must be a space character between the question mark and the additional parameter. For example: SENS1:CORR:WAV? MIN

Measurement Systems

The PM103E supports four different general measurement modes. The following section provides a detail description of these modes. Not all modes are supported by all Sensors or operating modes.

CW vs Peak mode

The PM103E supports two different measure modes for different sensor types. In CW mode the discrete input signal is sampled fixed at 100 kHz. Every millisecond the device processes the last 100 sample results. In CW mode the device outputs a constant measurement result stream regardless of input signal. This mode is supported for Photodiodes.

In Peak mode the PM103E internally samples the signal at 400 kHz and searches for peaks in the digitalized input data. Only peak values and their relative timestamp are forwarded to the calculation process. In peak mode, the measure stream output rate depends on the input signal repetition rate. If there are no peaks in input signal or the power meter is not configured appropriately no peaks are detected and there are no results to query. This mode is supported for Photodiodes and Pyroelectric sensors.

The Peak mode supports an overshoot filter for modulated signals. Use the command SENSe#:PEAKdetector:FILTer to enable/disable the overshoot filter. For sinus or triangle shaped signals disable the filter. It should be used for rectangular shaped signals where the flat top should be detected as peak and not the possible overshoot of the rising edge.

Slow Measurement System

The "Slow Measurement System" is available for all sensor types and measurement modes. It returns values up to a frequency of 1kHz. The command SENS:AVER can be used to scale down the update rate by formula MeasFreq = 1000/AVG in CW mode. In Peak mode the averaging is not used. Averaging is considered for the following units only:

  • POW: Photodiode: Sensor power in W or dBm (See command SENSe:POWer:UNIT)
  • ENER: Pyro: Sensor energy in Joule
  • CURR: Photodiode: Sensor current in Ampere
  • VOLT: Pyro: Sensor voltage in Volt
  • PDEN: Photodiode: Sensor power density in W/cm²
  • EDEN: Pyro: Sensor energy density in J/cm²

All other units have their own independent update rate.

In peak mode the output depends on the repetition rate of the input signal. For repetition rates larger than 1 kHz, all peaks within the recent millisecond will get averaged. Use the "Fast Measurement Stream" to access single pulses in that case.

The SCPI commands for this measurement system are part of the MEAS group. This group contains some convenience commands. These commands internally call a predefined SCPI command sequence. Starting with the well-known MEAS? convenience command:

  1. ABOR
    Aborts any ongoing measurement
  2. CONF
    Configures measurement channel for power measurement
  3. INIT
    Starts a new measurement
  4. FETC?
    Blocks until measurement is complete and returns the result

If the same unit should be read multiple times in a row, reconfiguring the channel again and again is not necessary. Therefore, use the READ? convenience command instead:

  1. ABOR
    Aborts any ongoing measurement
  2. INIT
    Starts a new measurement
  3. FETC?
    Blocks until measurement is complete and returns the result

The following diagram shows two examples of SCPI command sequences for the Slow Measure System. There is no timing constraint between the single SCPI commands. Timeline is from left to right.

The first example shows a 1 kHz measurement scenario with an averaging of 1. INIT starts a new measurement. Once the measurement is complete, the result is stored internally until FETC? is called. If FETC? is called multiple times in a row the same result will be returned as no new measurement has been started.

The second example show a 500 Hz measurement scenario with an averaging of 2. INIT starts the new measurement. Before the measurement result is ready FETC? is already called. As the measurement is still ongoing in the background the FETC? blocks the entire remote interface until measurement has been taken.

Once you send a FETC?, MEAS? or READ? command, the interface is blocked until the measurement has been taken. For Photodiode in CW mode sensors these commands work reliable except for very high averaging. High averaging values result in a long block time when waiting for the measurement to terminate. Continue reading to get a solution for this special case.

For Pyroelectric sensors and Photodiode sensors in Peak mode the blocking of FETC?, MEAS? or READ? might be a problem. For constant pulse repetition rates > 0.5 Hz the commands work reliable. But when working with low pulse rates or even single pulse signals it is very likely to end up in a communication timeout of the USB, Ethernet or serial interface. The PM103E internally blocks 10 seconds before the internal FETC? function terminates. Within this time do not send an ABOR or any other SCPI command. Please do not enlarge the communication interface timeout to solve this problem but continue reading.

  • Photodiode: AVG > 2000
  • Pyroelectric, Photodiode in Peak mode: Slow pulse repetition rates < 0.5 Hz
  • Pyroelectric, Photodiode in Peak mode: Single pulsed signals

Once a new measurement has been started by INIT, checking for completion can be done by the command FETC:STAT? If this command returns 1, FETC? can be called with providing an instant result. If the FETC:STAT? returns 0, continue to ask for completion or abort the measurement if not longer relevant by using the ABOR command. Do not use the convenience function in these cases.

Another configuration of the Slow Measurement system is the continuous measurement. For this mode there are no convenience functions. Use the following sequence for continuous mode:

  1. ABOR
    Aborts any previous ongoing measurement
  2. CONF:POW
    Configure a power measurement for example
  3. INIT:CONT
    Start continuous measurement
  4. FETC?
    Returns the last recent measurement result
  5. Delay()
    Delay between reading results
  6. FETC?
    Returns the last recent measurement result
  7. Delay()
    Delay between reading results
  8. ...

Keep in mind in continuous configuration there is no synchronization between the command and the measurements. If FETC? is called faster than the measurement update rate, the same result is read multiple times.

Fast Measurement Stream

The "Fast Measurement Stream" is available for Photodiodes in CW and peak mode and Pyroelectric sensors. The stream is accessed by the SCPI ACQ command group. The PM103E samples the analogue frontend at 100 kHz. For CW mode this results in 100 000 Samples per Second. The Fast Measurement Stream allows to accessing this data stream. There is no averaging available here. The data is stored in a queue allowing to buffer data for 10 ms. If the queue is full, new data will be lost. To ensure no data is lost, the reading needs to be as fast as possible. The data transfer of the ACQuire:FAST SCPI commands are binary to minimize transfer time and data conversion. The response has the following little endian byte order:

Every millisecond the PM103E adds a list of up to 100 tuples (time, measurement) to the queue as a non-dividable package. For Pyroelectric sensors or Photodiodes in Peak mode the Fast Measurement Stream is a list of all peak energy or power values found. The signal pulse repetition rate has to be smaller than 98 kHz.

All samples are tuples of relative time as 32-bit unsigned integer and measurement value as IEEE compliant float. The timestamp allows plotting the data, even if some data of stream is lost. Data loss is likely at some point because the PC is not always fast enough to query all data packages within 10 milliseconds. The wrapping 32-bit microsecond counter timestamp is relative and is not a wall time. This timestamp should be used to calculate the delta between the samples. For CW mode the delta is 10 us for sporadic pulse signals the delta is variable. To calculate the delta between the samples integer wrapping must be considered. The following example demonstrates the special case.

4294967292, 0.0001265

Use the following pseudo code to calculate the delta:

if(sample[n+1] > sample[n])
  delta = sample[n+1] - sample[n]
else
  delta = 0xffffffff - sample[n] + sample[n+1]

The "Fast Measurement Stream" does not support measuring result unit dBm. Changing the measurement range will interrupt the stream for multiple milliseconds, allowing the hardware to settle. If queue does not contain any data, ACQ:FAST:FETC? result length is read as zero without any further payload. Changing the unit of the stream will discard any previously queued measurement results.

A typical fast measurement stream command sequence looks like this:

  1. ACQ1:FAST:POW
    Configure fast measurement stream for power measurement
  2. ACQ1:FAST:RESET
    Reset the fast measurement stream
  3. ACQ1:FAST:FETC?
    Poll data of stream
  4. ACQ1:FAST:FETC?
    Poll data of stream
  5. ...

Scope Mode

The "Scope Mode" implements an oscilloscope like measurement method in the PM103E. It is only available for the CW mode for photodiode. Pyroelectric sensors are not supported by this measurement mode. An internal or external trigger starts a measurement to a fixed size internal buffer. Once the buffer is filled completely, the measurement stops automatically. Data remains valid in the buffer until the measurement gets started again. The buffer of the PM103E stores always 10 000 samples in total. The maximal sample rate in scope mode is 100 kHz. With the 10 000 samples this results in a total time of 100 ms with highest resolution. For longer time intervals the averaging of scope must be used. If averaging is set to 2 the scope buffer is filled at 50 kHz resulting in a total time of 200 ms.

Before starting a scope measurement for optical power reading ensure:

  • Set manual range (Disables auto ranging of PM103E automatically), command: SENSe#:POWer[:DC]:RANGe[:UPPer]
  • Set full bandwidth for Photodiode sensors (no analog filter), command: INPut#[:PDIode]:FILTer[:LPASs][:STATe] 0
  • Choose CW mode for Photodiode sensors, command: SENSe#:FREQuency:MODE CW
  • Set trigger threshold for internal hardware triggered signal, command: SENSe:PEAK (Not applicable for external triggering)
  • To select external trigger configure SOUR:DIG:PIN1:FUNC IALT. To switch back to internal trigger select other pin function.

The scope mode supports two trigger types. First the simpler software trigged Scope mode. Here the sample buffer gets filled directly after receiving the INIT SCPI command. If FETC? is called before the buffer has been filled, the command will block. An averaging pre-scaler allows to sample longer periods with less resolution. Any incoming HW triggers are ignored.
Second the more complex hardware triggered scope mode. The following diagram visualizes the mode. Time flows from left to right.

After sending ABOR to cancel any previous ongoing measurement. You can select two different triggers sources. By default the frequency count threshold is used as internal trigger condition. Every time the signal becomes larger than the configured threshold (See command SENSe:PEAK), the internal trigger fires. To use an external trigger you have to configure GPIO1 pin for alternate input (See command SOUR:DIG:PIN1:FUNC). The external trigger reacts on rising edges. During configuration the trigger is not armed. After receiving the first CONF:ARR:HWT the PM103E directly starts storing measurement results in the pre-trigger buffer. The CONF:ARR:HWT sets the averaging pre-scaler to allow longer time periods and the trigger horizontal position (hPos) to allow the scope to show also part of the signal before the trigger condition. The diagram shows an example for averaging = 1 and hPos = 100. The measurement is not yet started, and the trigger is not armed. Any incoming trigger condition is ignored.

With the INIT command the trigger becomes armed. The next trigger condition will start the scope measurement if the pre-trigger buffer is filled already. If not, the condition is skipped as well. When the trigger is valid the scope continues storing the measurement results in the post-trigger buffer and unarms the trigger automatically until a future INIT invalidates the previous buffer and starts a new measurement. Any incoming FETC? or FETC:ARR? during an ongoing measurement blocks until the buffer is filled completely. If the post-trigger buffer is filled completely, the PM103E continues measuring in the shadow pre-trigger buffer for the next measurement in background. The results of pre-trigger and post-trigger buffers stay valid and can be read with FETC? or FETC:ARR? at any time and speed.

The entire scope calculates in samples not time. The averaging allows to transfer samples to time at any time. The hPos is only allowed to be between 0 and 5000. The number of samples in post-trigger buffer always equals 10000 - hPos. The following example shows how to convert samples into time domain:

CONF:ARR:HWT 5,150

  • Scope sample period time: Avg = 5 -> 100 kHz / 5 = 20 kHz = 50 us
  • Time before trigger condition: hPos = 150 Samples -> 150 * 50 us = 7,5 ms
  • Total time in buffer: 10000 * 50 us = 500 ms
  • Time after trigger condition = 500 ms – 7.5 ms or (10000 – 150) * 50 us = 492.5 ms

  • A trigger condition can be forced by command TRIGer:ARRay:FORCe in hardware triggered Scope mode
  • Scope mode is not available for Pyroelectric sensors or Photodiode sensors in Peak mode
  • Reading data in binary format by FETC:ARR? is much faster than the normal FETC? command
  • There are more flags in the STAT:OPER:COND? register showing the state of the scope buffer
  • For short scope measurements also the related convenience commands can be used: MEAS:ARR:HWT?, MEAS:ARR? or READ:ARR?

Burst Mode

The "Burst Mode" allows storing hardware triggerd measure results in an device internal buffer to be read out later. It uses the same memory than the "Scope Mode". It is only available for the CW mode for photodiode. Pyroelectric sensors are not supported by this measurement mode. Multiple internal or external trigger conditions start a measure burst sequence if not started already. The trigger repetition rate must be larger or equal 100 us. Normally the external trigger input of GPIO pin 0 is used. The device memory allows storing up to 15000 samples. Every sequence is defined like this:

  • Optional initial delay ("Init Delay") before starting measurement burst
  • Averaged burst measurement 1
  • Optional averaged burst measurement 2
  • Optional averaged burst measurement 3
  • ...

The optional initial delay allows to postpone the burst sequence after the trigger condition in a 10 us time grid. If not required set to 0 to start sequence immediately. The sequence itself always contains at least one but less than 5000 averaged measurements. If the burst sequence stops the next trigger will issue an identical next burst sequence. If there is no more free memory in the device, data is discarded. An optional averaging enables the burst sequence to measure longer time periods with the same amount of stored samples. To store a single measure result with highest resolution for every trigger use Init Delay 0, Averaging 1 and a Burst Length of 1.

Before starting a burst measurement for optical power reading ensure:

  • Set manual range (Disables auto ranging of PM103E automatically), command: SENSe#:POWer[:DC]:RANGe[:UPPer]
  • When measuring pulsed or alternating signals, set full bandwidth for Photodiode sensors (no analog filter), command: INPut#[:PDIode]:FILTer[:LPASs][:STATe] 0
  • Choose CW mode for Photodiode sensors, command: SENSe#:FREQuency:MODE CW
  • Set trigger threshold for internal hardware triggered signal, command: SENSe:PEAK (Not applicable for external triggering)
  • To select external trigger configure SOUR:DIG:PIN1:FUNC IALT. To switch back to internal trigger select other pin function.

Every burst measurement point is a tuple of relative sample time followed by measurement value. The example shows a burst configuration with an Init Delay of 2 (20 us) and a Burst Count of 3 with an Averaging of 2 (20 us).

In the given example the device will idle for 20 us once the trigger fired before starting to store 3 samples in a row with an averaging of 20 us. Any trigger during this time will be postponed to the end of the burst. In that case the next burst to starts directly after the previous. Ensure the trigger repetition rate is >= 100 us (10 kHz). As long there are rising edge trigger conditions and there are less than 15000 samples already stored the burst sequence the burst sequence gets restarted. Once the measurement is complete call ABOR to deactivate the burst mode. The previous results stay valid until next INIT. Use FETC:ARR:LEN? to get the amount of samples stored. Then query the results using FETC? or the faster but binary FETC:ARR? command multiple times. Calling any of these commands during ongoing measurement result in an error. You may call FETC:ARR:LEN? during measurement to see how many results are already stored.

The entire burst calculates in samples not time. The averaging allows to transfer samples to time at any time. The following example shows how to convert a parameter set of CONF:ARR:BUR 5, 2, 3 into time domain:

  • Init Delay time: InitDelay = 5, 10 us * 5 = 50 us
  • Averaging time: Averaging = 3, 10 us * 3 = 30 us
  • Burst sequence time: (InitDelay + SequcenceCount * Averaging) * 10 us = 110 us

Notes:

  • A trigger condition can be forced by command TRIGer:ARRay:FORCe in Burst mode
  • Burst mode is not available for Pyroelectric sensors or Photodiode sensors in Peak mode
  • Reading data in binary format by FETC:ARR? is much faster than the normal FETC? command

Autoranging

In "CW" mode, the autoranging feature increases the range (making it less sensitive) if the measured value is within the upper or lower 3% of the current measurement range for five consecutive readings. Conversely, the power meter reduces the range (increasing sensitivity) if the measured value falls within the interval of the next more sensitive gain range for 100 consecutive measurements. The system employs a 10% hysteresis for the more sensitive gain to prevent frequent toggling between ranges.

Note that when the range changes, the hardware requires up to 20 milliseconds to stabilize, during which time the power meter will not provide any measurements.

In Peak mode, autoranging is only effective for repetitive pulsed signals with a repetition rate greater than 5 Hz. If no pulses are detected for one second, the power meter uses a peak finder algorithm to determine a peak and threshold level. Once a valid peak is detected, the same autoranging method as in CW mode is applied. To expedite the process, the meter will adjust the range up or down after two consecutive out-of-range values. Similar to CW mode, changing the range will cause a brief stabilization pause, during which pulses are ignored.

Additionally, in Peak mode, the autoranging feature adjusts the trigger level to be approximately 8% below the peak maximum. Thus, the energy or power changes between consecutive pulses must be below 5%. Manual adjustment of the threshold is not allowed while autoranging is enabled. After autoranging is disabled in Peak mode, the threshold remains at the last used value until manually changed. The current autoranging threshold can be queried at any time using the SCPI command SENSe:PEAK?.

JSON Format

The power meter allows to export and import all measurement related parameters in JSON format. To simplify JSON transport via SCPI, the JSON response does not contain any line breaks or other formats characters. The JSON strings are not designed to change the measurement parameters. The intention is to provide a simple store and load parameters functionality. It is possible to change the sensors within the sensor type family. For example a photodiode might be replaced by another photodidode. Importing the photodiode parameters for a thermopile sensor will cause an error. If the sensor is changed during export and import it might be necessary to adapt some parameters to fit the new sensor. For exammple the wavelength needs to be adjusted. This adaption is optional on request. For the import the JSON string needs to be split into multiple SCPI command calls as the maximal request length is limited. The JSON parts are buffered in the device. Once the entire string has been transmitted the power meter will parse the JSON and update the parameters. SCPI Registers

      
The SCPI system defines a status reporting system based on status register sets. Every state or event is mapped to a single bit in the values of the register sets. 
      The PM103E defines the following register sets: 

      
    
        
  • AUXilary: Auxiliary system monitoring like shutter or fan.
    • 
        
  • OPERation: Measurement state monitoring.
    • 
        
  • QUESTionalbe: Measure result reliability status monitoring.
    • 
        
  • STDandard Byte: SCPI global system status byte.
    • 
      


      
Every register set contains 5 registers.

      
    
        
  • CONDition: The Condition register monitors the actual state. For example if a sensor is currently connected or not.
    • 
        
  • PTRansition: The Positive Transition register masks bit changes from 0 to 1 of the Condition register to the Event register.
    • 
        
  • NTRansition: The Negative Transition register masks bit changes from 1 to 0 of the Condition register to the Event register.
    • 
        
  • EVENT: The Event register shows changes of the Condition register. Some conditions only last for a short moment and are otherwise hard to detect.
    • 
        
  • ENABle: The Enable register masks if changes of the Event register are forwarded to a higher level hierarchy Condition register.
    • 
      

      
The following image visualizes the logical relation between the registers.

      
      
The Condition register always reflects the current state. For example if a sensor is connected or not. The Positive Transition register masks if a
      certain condition becomes true (bit changes from 0 to 1) to set the bit at the same position in the Event register. For example to set the Event register 
      bit to 1 if the sensor got connected. 
The Negative Transition register tests if condition becomes false to update the Event register. For example to 
      set event register bit to 1 if the sensor got disconnected. The Enable register masks if changes of the Event register should be reported to the 
      higher level hierarchy Condition register. The register sets are logically chained. The following section will provide further details about the chaining.

      
All registers except the STB register group are 16 bit wide. The following paragraph gives a short description of bit mask in general. Bit mask use the binary
      representation of numbers to store boolean states (true, false) or (active, inactive). This is a memory saving way to store multiple conditions at once.
      If you look on a decimal number 9 for example the binary representation is 1001 = 1 * 2 ^ 3 + 1 * 2 ^ 0. This bit mask means condition at bit position 1 and 4 are true. 
      All other conditions are false as all other bits are set to 0.

      To test if a certain bit is set to 1 you can either use bit logic operations or use generic mathematic operations. Bit logic operations are available in almost every 
      programming language. In Python you would write something like condition = bitmask & (1 << position).

      If you want to use standard math you can use the following formula: condition = round_down(bitmask / (2 ^ bitPos)) modulus 2.

      
The register sets are linked together in an hirarchical way. This enables you to check for a summarized condition only. If the 
      highlevel condition becomes true you can than go down the hierarchy to query the details causing the event.


      
      
The STAT:OPER status register set contains state information about the instrument’s normal
      operation. The register set might report to the SCPI Standard Byte register bit 7 if 
      STAT:OPER:ENAB bit mask is set active.
  

      
     
The QUEStionable status register set contains bits which give an indication of the quality of
      various aspects of the measurement signal. The register set might report to the SCPI Standard 
      Byte register bit 3 if STAT:OPER:ENAB bit mask is set active.

     
      
      
The AUXilary register set contains state informatino about the instrument auxiliary like the 
      NTC or power supply. The register set might report to the SCPI Standard 
      Byte register bit 0 if STAT:OPER:ENAB bit mask is set active.

 val = tuple[0]

 if(curr_t > prev_t)
