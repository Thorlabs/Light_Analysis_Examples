This is the description of the Thorlabs PM60 Series remote interface. This single channel Powermeter can be remotely controlled using the SCPI (Standard Commands for Programmable Instruments) interface.

In 1975, the IEEE standardized a bus developed by Hewlett-Packard originally called HPIB (Hewlett-Packard Interface Bus), later changed to GPIB (General Purpose Interface Bus). The standard was called IEEE 488 (IEEE 488.1) and it defined mechanical aspects of the bus. The later IEEE 488.2 defined its protocol properties. What was missing were set of rules between the manufacturers on commands to control the instruments. Sometimes these varied even between different models from the same manufacturer.

In 1990 SCPI Consortium released the first SCPI standard as an additional layer for the IEEE-488.2 standard.

SCPI commands are ASCII strings, which are sent to instrument over the physical communication layer. They can perform:

  • Set operations, for example the *RST command (resetting the instrument).
  • Query operations, for example the *IDN? query (querying the instrument’s identification string).

Some SCPI commands exist as both set commands and query commands. An example is an Powermeter's averaging command SENS:AVER. You can set it with the SCPI command SENS:AVER 2, and also query its current value with the SENS:AVER?.

The format mentioned in this manual e.g.: SENSe#:AVERage[:COUNt] <pres> is called canonical form. Here are the most important rules to remember:

  • The parts within square brackets are not mandatory and can be left out.
  • All # within the command represent a suffix positive integer number starting at 1. If there is only one suffix you may skip it because default is applied internally. If there are two or more suffix it is mandatory to specify all.
  • The capital letter parts are mandatory; the small letters can be omitted. This is called short form. An example of the above command in the short form is SENS1:AVER 3. You can use either the short form, or the long form SENSE1:AVERAGE 3, but nothing in between, e.g. SENS1:AVERA 3.
  • The SCPI commands are case-insensitive. You can also use the short form sens1:aver 3 or or the long form sense1:average 3
  • Combine multiple commands into one string using selmicolon ';'. For example, a combined string of SENS1:AVER 3 and SENS1:CORR:WAV 850 is SENS1:AVER 3;CORR:WAV 850. Notice that the second command does not have the SENS1: part. The reason is, that the command tree path does not change within one string. If you want to reset the command tree path to the root, use the colon character at the beginning of the second command: SENS1:AVER 3;:SENS2:CORR:WAV 850.
  • Create query forms by adding a question mark, mostly to the end: SENS1:AVER? Sometimes there is an additional parameter placed after the question mark. There must be a space character between the question mark and the additional parameter. For example: SENS1:CORR:WAV? MIN

Measurement Systems

The PM60 supports three different general measurement modes. The following section provides a detailed description of these modes. Not all modes are supported by all sensors or operating modes.

CW vs Peak mode

The PM60 supports two different measurement modes in general. In CW mode the discrete input signal is sampled fixed at 100 kHz. Every millisecond the device processes the last 100 sample results. In CW mode the device outputs a constant measurement result stream regardless of input signal. This mode is supported for Photodiodes or Thermopiles.

In Peak mode the PM60 internally samples the signal at 400 kHz and searches for peaks in the digitalized input data. Only peak values and their relative timestamp are forwarded to the calculation process. In peak mode, the measurement stream output rate depends on the input signal repetition rate. If there are no peaks in input signal or the powermeter is not configured appropriately no peaks are detected and there are no results to query. This mode is supported for Photodiodes only.

The Peak mode supports an overshoot filter for modulated signals. Use the command SENSe#:PEAKdetector:FILTer to enable/disable the overshoot filter. For sinus or triangle shaped signals disable the filter. It should be used for rectangular shaped signals where the flat top should be detected as peak and not the possible overshoot of the rising edge.

Slow Measurement System

The "Slow Measurement System" is available for all sensor types and measurement modes. It returns values up to a frequency of 1kHz. The command SENS:AVER can be used to scale down the update rate by formula MeasFreq = 1000/AVG in CW mode. In Peak mode the averaging is not used. Averaging is considered for the following units only:

  • POW: Photodiode/Thermopile: Sensor power in W or dBm (See command SENSe:POWer:UNIT)
  • CURR: Photodiode: Sensor current in Ampere
  • VOLT: Thermopile: Sensor voltage in Volt
  • PDEN: Photodiode: Sensor power density in W/cm²

All other units have their own independent update rate.

In peak mode the output depends on the repetition rate of the input signal. For repetition rates larger than 1 kHz, all peaks within the recent millisecond will get averaged.

The SCPI commands for this measurement system are part of the MEAS group. This group contains some convenience commands. These commands internally call a predefined SCPI command sequence. Starting with the well-known MEAS? convenience command:

  1. ABOR
    Aborts any ongoing measurement
  2. CONF
    Configures measurement channel for power measurement
  3. INIT
    Starts a new measurement
  4. FETC?
    Blocks until measurement is complete and returns the result

If the same unit should be read multiple times in a row, reconfiguring the channel again and again is not necessary. Therefore, use the READ? convenience command instead:

  1. ABOR
    Aborts any ongoing measurement
  2. INIT
    Starts a new measurement
  3. FETC?
    Blocks until measurement is complete and returns the result

The following diagram shows two examples of SCPI command sequences for the Slow Measurement System. There is no timing constraint between the single SCPI commands. Timeline is from left to right.

The first example shows a 1 kHz measurement scenario with an averaging of 1. INIT starts a new measurement. Once the measurement is complete, the result is stored internally until FETC? is called. If FETC? is called multiple times in a row the same result will be returned as no new measurement has been started.

The second example shows a 500 Hz measurement scenario with an averaging of 2. INIT starts the new measurement. Before the measurement result is ready FETC? is already called. As the measurement is still ongoing in the background the FETC? blocks the entire remote interface until measurement has been taken.

Once you send a FETC?, MEAS? or READ? command, the interface is blocked until the measurement has been taken. For Photodiode in CW mode and Thermopile sensors these commands work reliable except for very high averaging. High averaging values result in a long block time when waiting for the measurement to terminate. Continue reading to get a solution for this special case.

For Photodiode sensors in Peak mode the blocking of FETC?, MEAS? or READ? might be a problem. For constant pulse repetition rates > 0.5 Hz the commands work reliable. But when working with low pulse rates or even single pulse signals it is very likely to end up in a communication timeout of the USB or bluetooth interface. The PM60 internally blocks 10 seconds before the internal FETC? function terminates. Within this time do not send an ABOR or any other SCPI command. Please do not enlarge the communication interface timeout to solve this problem but continue reading.

  • Photodiode in Peak mode: Slow pulse repetition rates < 0.5 Hz
  • Photodiode in Peak mode: Single pulsed signals

Scope Mode

The "Scope Mode" implements an oscilloscope like measurement method in the PM60. It is only available for the CW mode for photodiode and thermopile sensors. An internal or external trigger starts a measurement to a fixed size internal buffer. Once the buffer is filled completely, the measurement stops automatically. Data remains valid in the buffer until the measurement gets started again. The buffer of the PM60 stores always 10 000 samples in total. The maximal sample rate of the two measurement channels in scope mode is 100 kHz. With the 10 000 samples this results in a total time of 100 ms with highest resolution. For longer time intervals the averaging of scope must be used. If averaging is set to two the scope buffer is filled at 50 kHz resulting in a total time of 200 ms.

Before starting a scope measurement for e.g. optical power reading ensure:

  • Set manual range (Disables auto ranging of PM60 automatically), command: SENSe#:POWer[:DC]:RANGe[:UPPer]
  • Set full bandwidth for Photodiode sensors (no analog filter), command: INPut#[:PDIode]:FILTer[:LPASs][:STATe] 0
  • Choose CW mode for Photodiode sensors, command: SENSe#:FREQuency:MODE CW
  • Disable acceleration for thermopile sensors, command: INPut#:THERmopile:ACCelerator:AUTO 0
  • Set trigger threshold for internal hardware triggered signal, command: CONFigure#:ARRay:HWTrig (Not applicable for external triggering)

The scope mode is part of the MEAS SCPI command group. The basic command sequence looks like this:

  1. ABOR
    Abort any ongoing measurement
  2. CONF:ARR:CHA
    Configure unit for channel.

  3. CONF:ARR 1
    Configure software Scope mode with averaging 1 OR
    CONF1:ARR:HWT 1,100
    Configure hardware Scope mode with averaging 1 and hpos 100 for internal trigger.

  4. INIT
    Starts a scope measurement
  5. FETC:STAT?
    Scope buffer filled? If not keep on asking. For timeout send ABOR finally
  6. FETC:STAT?
    Scope buffer filled? If not keep on asking. For timeout send ABOR finally
  7. ...
  8. FETC:STAT?
    Scope buffer filled returned true.
  9. FETC? 0
    Read scope buffer content at index 0
  10. FETC? 1
    Read scope buffer content at index 1
  11. ...
  12. FETC? 9999
    Read scope buffer content at index 9999

Once configured, a new scope measurement is taken with the following sequence:

  1. INIT
    Starts a scope measurement. Uses previous configuration. Invalidates previous buffer content!
  2. FETC:STAT?
    Scope buffer filled?
  3. ...
  4. FETC:STAT?
    Scope buffer filled returned true.
  5. FETC? 0
    Read scope buffer content at index 0
  6. FETC? 1
    Read scope buffer content at index 1
  7. ...
  8. FETC? 9999
    Read scope buffer content at index 9999

The scope mode supports two trigger types. First the simpler software trigged Scope mode. Here the sample buffer gets filled directly after receiving the INIT SCPI command. If FETC? is called before the buffer has been filled, the command will block. An averaging pre-scaler allows to sample longer periods with less resolution. Any incoming HW triggers are ignored.
Second the more complex hardware triggered scope mode. The following diagram visualizes the mode. Time flows from left to right.

After receiving the first CONF:ARR:HWT the PM60 directly starts storing measurement results in the pre-trigger buffer. The CONF:ARR:HWT sets the averaging pre-scaler to allow longer time periods and the trigger horizontal position (hPos) to allow the scope to show also part of the signal before the trigger condition. The diagram shows an example for averaging = 1 and hPos = 100. The measurement is not yet started, and the trigger is not armed. Any incoming trigger condition is ignored.

With the INIT command the trigger becomes armed. The next trigger condition will start the scope measurement if the pre-trigger buffer is filled already. If not, the condition is skipped as well. When the trigger is valid the scope continues storing the measurement results in the post-trigger buffer and unarms the trigger automatically until a future INIT invalidates the previous buffer and starts a new measurement. Any incoming FETC? or FETC:ARR? during an ongoing measurement blocks until the buffer is filled completely. If the post-trigger buffer is filled completely, the PM60 continues measuring in the shadow pre-trigger buffer for the next measurement in background. The results of pre-trigger and post-trigger buffers stay valid and can be read with FETC? or FETC:ARR? at any time and speed.

The entire scope calculates in samples not time. The averaging allows to transfer samples to time at any time. The hPos is only allowed to be between 0 and 5000. The number of samples in post-trigger buffer always equals 10000 - hPos. The following example shows how to convert samples into time domain:

CONF1:ARR:HWT 5,150

  • Scope sample period time: Avg = 5 -> 100 kHz / 5 = 20 kHz = 50 us
  • Time before trigger condition: hPos = 150 Samples -> 150 * 50 us = 7,5 ms
  • Total time in buffer: 10000 * 50 us = 500 ms
  • Time after trigger condition = 500 ms – 7.5 ms or (10000 – 150) * 50 us = 492.5 ms

Notes:

  • A trigger condition can be forced by command TRIGer:ARRay:FORCe in hardware triggered Scope mode
  • Scope mode is not available for Photodiode sensors in Peak mode
  • Reading data in binary format by FETC:ARR? is much faster than the normal FETC? command
  • The timebase is a relative wrapping uint32 microsecond counter. Read timebase section carefully.
  • There are more flags in the STAT:OPER:COND? register showing the state of the scope buffer
  • For short scope measurements also the related convenience commands can be used: MEAS:ARR:HWT?, MEAS:ARR? or READ:ARR?

Burst Mode

The "Burst Mode" allows storing hardware triggerd measurement results in an device internal buffer to be read out later. It uses the same memory than the "Scope Mode". It is only available for the CW mode for photodiode and thermopile. Multiple internal or external trigger conditions start a measurement burst sequence if not started already. The trigger repetition rate must be larger or equal 100 us. The hardware trigger is not available on all models of the PM6x series. Please refer to the device data sheet for detail pinning information. The device memory allows storing up to 15000 samples. Every sequence is defined like this:

  • Optional initial delay ("Init Delay") before starting measurement burst
  • Averaged burst measurement 1
  • Optional averaged burst measurement 2
  • Optional averaged burst measurement 3
  • ...

The optional initial delay allows to postpone the burst sequence after the trigger condition in a 10 us time grid. If not required set to 0 to start sequence immediately. The sequence itself always contains at least one but less than 5000 averaged measurements. If the burst sequence stops the next trigger will issue an identical next burst sequence. If there is no more free memory in the device, data is discarded. An optional averaging enables the burst sequence to measure longer time periods with the same amount of stored samples. To store a single measurement result with highest resolution for every trigger use Init Delay 0, Averaging 1 and a Burst Length of 1.

Before starting a burst measurement for optical power reading ensure:

  • Set manual range (Disables auto ranging of PM60 automatically), command: SENSe#:POWer[:DC]:RANGe[:UPPer]
  • When measuring pulsed or alternating signals, set full bandwidth for Photodiode sensors (no analog filter), command: INPut#[:PDIode]:FILTer[:LPASs][:STATe] 0
  • Choose CW mode for Photodiode sensors, command: SENSe#:FREQuency:MODE CW
  • Disable acceleration for thermopile sensors, command: INPut#:THERmopile:ACCelerator:AUTO 0
  • Set trigger threshold for internal hardware triggered signal, command: CONFigure#:ARRay:HWTrig (Not applicable for external triggering)

The burst mode is part of the MEAS SCPI command group. The basic command sequence looks like this:

  1. ABOR
    Abort any ongoing measurement
  2. CONF:ARR:CHA
    Configure unit for channel 1.
  3. CONF2:ARR:BUR 1,2,3
    Configure hardware GPIO triggered burst mode with initDelay = 1, BustCount = 2 and Averaging = 3.
  4. INIT
    Starts a burst measurement
  5. Trigger is active and burst sequences are stored in device buffer
  6. ABOR
    Stops burst measurement. Triggers are not longer observed
  7. FETC:ARR:LEN?
    Reads amount of samples in buffer
  8. FETC? 0
    Reads first sample of burst sequence
  9. FETC? 1
    Reads second sample of burst sequence
  10. ...

Once configured, a new burst measurement is taken with the following sequence:

  1. INIT
    Starts a burst measurement. Uses previous configuration
  2. Trigger is active and burst sequences are stored in device buffer
  3. ABOR
    Stops burst measurement. Triggers are not longer observed
  4. FETC:ARR:LEN?
    Reads amount of samples in buffer
  5. FETC? 0
    Reads first sample of burst sequence
  6. FETC? 1
    Reads second sample of burst sequence
  7. ...

The following diagram visualizes the mode in more detail. Time flows from left to right.

During configuration of Burst Mode all trigger conditions are ignored. After configuring the unit for the single channels using for example CONF:ARR:CHAN you configure the Burst mode in general. The CONF2:ARR:BUR command suffix selects the trigger input. By default the frequency count threshold of channel 1 is used as internal trigger condition. In most cases it makes sense to use GPIO1 as trigger input. The external 3.3V level trigger input react on rising edges. The internal trigger of channel 1 fires when the signal becomes larger than the configured threshold (See command SENSe:PEAK). The CONF2:ARR:BUR parameters define the burst sequence to store after trigger condition. If you call INIT afterwards the trigger is armed and start a new burst measurement sequence. The burst results are stored without any separator in the burst memory. Every burst measurement point is a tuple of relative sample time and value for channel 1. The example shows a burst configuration with an Init Delay of 2 (20 us) and a Burst Count of 3 with an Averaging of 2 (20 us).

In the given example the device will idle for 20 us once the trigger fired before starting to store 3 samples in a row with an averaging of 20 us. Any trigger during this time will be postponed to the end of the burst. In that case the next burst to starts directly after the previous. Ensure the trigger repetition rate is >= 100 us (10 kHz). As long there are rising edge trigger conditions and there are less than 15000 samples already stored the burst sequence the burst sequence gets restarted. Once the measurement is complete call ABOR to deactivate the burst mode. The previous results stay valid until next INIT. Use FETC:ARR:LEN? to get the amount of samples stored. Then query the results using FETC? or the faster but binary FETC:ARR? command multiple times. Calling any of these commands during ongoing measurement result in an error. You may call FETC:ARR:LEN? during measurement to see how many results are already stored.

The entire burst calculates in samples not time. The averaging allows to transfer samples to time at any time. The following example shows how to convert a parameter set of CONF2:ARR:BUR 5, 2, 3 into time domain:

  • Trigger source: 2 = GPIO or 1 = Channel 1
  • Init Delay time: InitDelay = 5, 10 us * 5 = 50 us
  • Averaging time: Averaging = 3, 10 us * 3 = 30 us
  • Burst sequence time: (InitDelay + SequcenceCount * Averaging) * 10 us = 110 us

Notes:

  • A trigger condition can be forced by command TRIGer:ARRay:FORCe in Burst mode
  • Burst mode is not available for Photodiode sensors in Peak mode
  • Reading data in binary format by FETC:ARR? is much faster than the normal FETC? command
  • The timebase is a relative wrapping uint32 microsecond counter. Read timebase section carefully.

Relative Timebase

To calculate the time delta between fast sample data a relative wrapping 32-bit microsecond counter is used. This timestamp is relative and is not a wall time. When calculating the time delta, integer wrapping must be considered. The following example demonstrates the special case. Time delta is 10 microseconds here.

4294967282, 0.0001233
4294967292, 0.0001265
                  6, 0.0001274
                16, 0.0001254

Use the following pseudo code to calculate the delta in Python:

if sample[n+1] > sample[n]:
  delta = sample[n+1] - sample[n]
else
  delta = 0xffffffff - sample[n] + sample[n+1]

In C you can use the following code:

int32_t delta_t = (int32_t)sample[n+1] - (int32_t)sample[n];

Binary data format

For some SCPI commands binary data is transferred instead of ASCII text data. This minimizes data traffic and speeds up data parsing. The device uses Little Endian byte order. For example for the FETC:ARR? command you will always receive up to 100 tuples at once. The binary tuple list response of FETC:ARR? has the following meaning:

The first 4 bytes integer contains the amount of tuples in the list. Every tuple starts with a 4 byte integer relative microsecond timestamp followed by a 4 byte IEEE compliant float. Do not scan for any separator between the data or a line end. Use read bytes command to read tuple list length first. Then calculate the payload length and read tuple list afterward.

The following code gives an example to read and parse the binary tuple data in Python:

inst.write('FETC:ARR? 0,100')
vals = inst.read_bytes(4)
length = struct.unpack('<I', bytearray(vals[0 : 0 + 4]))[0]

vals = inst.read_bytes(8 * length)

res = []
for i in range(0, length, 8):
   reltime = struct.unpack('<I', bytearray(vals[i:i + 4]))[0]
   value   = struct.unpack('<f', bytearray(vals[i + 4:i + 8]))[0]
   res.append([reltime, value])

JSON Format

The powermeter allows to export and import all measurement related parameters in JSON format. To simplify JSON transport via SCPI, the JSON response does not contain any line breaks or other formats characters. The JSON strings are not designed to change the measurement parameters. The intention is to provide a simple store and load parameters functionality. It is possible to change the sensors within the sensor type family. For example a photodiode might be replaced by another photodidode. Importing the photodiode parameters for a thermopile sensor will cause an error. If the sensor is changed during export and import it might be necessary to adapt some parameters to fit the new sensor. For exammple the wavelength needs to be adjusted. This adaption is optional on request. For the import the JSON string needs to be split into multiple SCPI command calls as the maximal request length is limited. The JSON parts are buffered in the device. Once the entire string has been transmitted the powermeter will parse the JSON and update the parameters. SCPI Registers

      
The SCPI system defines a status reporting system based on status register sets. Every state or event is mapped to a single bit in the values of the register sets. 
      The PM60 defines the following register sets: 

      
    
        
  • AUXilary: Auxiliary system monitoring like shutter or fan.
    • 
        
  • OPERation: Measurement state monitoring.
    • 
        
  • QUESTionalbe: Measurement result reliability status monitoring.
    • 
        
  • STDandard Byte: SCPI global system status byte.
    • 
      


      
Every register set contains 5 registers.

      
    
        
  • CONDition: The Condition register monitors the actual state. For example if a sensor is currently connected or not.
    • 
        
  • PTRansition: The Positive Transition register masks bit changes from 0 to 1 of the Condition register to the Event register.
    • 
        
  • NTRansition: The Negative Transition register masks bit changes from 1 to 0 of the Condition register to the Event register.
    • 
        
  • EVENT: The Event register shows changes of the Condition register. Some conditions only last for a short moment and are otherwise hard to detect.
    • 
        
  • ENABle: The Enable register masks if changes of the Event registers are forwarded to a higher level hierarchy Condition register.
    • 
      

      
The following image visualizes the logical relation between the registers.

      
      
The Condition register always reflects the current state. For example if a sensor is connected or not. The Positive Transition register masks if a
      certain condition becomes true (bit changes from 0 to 1) to set the bit at the same position in the Event register. For example to set the Event register 
      bit to 1 if the sensor got connected. 
The Negative Transition register tests if condition becomes false to update the Event register. For example to 
      set event register bit to 1 if the sensor got disconnected. The Enable register masks if changes of the Event register should be reported to the 
      higher level hierarchy Condition register. The register sets are logically chained. The following section will provide further details about the chaining.

      
All registers except the STB register group are 16 bit wide. The following paragraph gives a short description of bit mask in general. Bit mask use the binary
      representation of numbers to store boolean states (true, false) or (active, inactive). This is a memory saving way to store multiple conditions at once.
      If you look on a decimal number 9 for example the binary representation is 1001 = 1 * 2 ^ 4 + 1 * 2 ^ 0. This bit mask means condition at bit position 1 and 4 are true. 
      All other conditions are false as all other bits are set to 0.

      To test if a certain bit is set to 1 you can either use bit logic operations or use generic mathematic operations. Bit logic operations are available in almost every 
      programming language. In Python you would write something like condition = bitmask & (1 << position).

The register sets are linked together in an hirarchical way. This enables you to check for a summarized condition only. If the 
      highlevel condition becomes true you can than go down the hierarchy to query the details causing the event.


      
      
The STAT:OPER status register set contains state information about the instrument’s normal
      operation. The register set might report to the SCPI Standard Byte register bit 7 if 
      STAT:OPER:ENAB bit mask is set active.
  

     
     
The QUEStionable status register set contains bits which give an indication of the quality of
      various aspects of the measurement signal. The register set might report to the SCPI Standard 
      Byte register bit 3 if STAT:OPER:ENAB bit mask is set active.

     
      
      
The AUXilary register set contains state informatino about the instrument auxiliary like the
